# AI-Native Workflow Templates

Opinionated, reusable templates and patterns for running AI coding agents at scale — focused on spec-kitty missions, multi-agent orchestration, prompt-cache discipline, and review-loop patterns. Each template is self-contained, documented, and ships with a runnable example you can copy into your own repo and adapt.

## Catalog

Fifty-two templates, grouped by what they do.

| [`templates/tool-call-deduplication`](templates/tool-call-deduplication/) | In-process, signature-hash deduplication of tool calls — the sibling of `tool-call-retry-envelope` for the case where the *agent itself* re-issues the same logical call within seconds (thought loops, partial-trace confusion). Caller mints no key; the host hashes `(tool_name, canonical(identity_args))` with a closed `identity_fields` allowlist so volatile metadata (request-ids, timestamps) does not defeat dedup. Floats in identity-args raise `CanonicalizationError` (silent precision loss is a correctness trap); dict-key order does not change the hash; lazy expiry on access keeps memory bounded. Stdlib-only `DedupCache(window_seconds, now_fn)` with deterministic injected clock. Two worked examples — agent loops and a 50ms-later re-issue of `read_file` is served from cache with `original_call_id` preserved for the audit link, and an `identity_fields=["query","limit"]` allowlist correctly absorbs two calls that differ only in `request_id`/`now` while `limit=25→50` correctly re-executes, plus an 11s clock advance evicts past the 10s window and a literal `0.875` in identity-args is rejected with a JSON pointer to the offending field. Composes with `tool-call-retry-envelope` (cross-process, caller-keyed twin), `agent-decision-log-format` (log every `verdict=use_cached` with `original_call_id`), and `tool-permission-grant-envelope` (dedup runs *after* the grant decision so denied calls are never cached). |
| [`templates/streaming-chunk-reassembler`](templates/streaming-chunk-reassembler/) | Pure, single-stream reassembler for streamed LLM/tool chunks delivered out of order, duplicated, or with persistent gaps. `accept(chunk) -> list[chunk]` returns only the chunks that became deliverable in strict `seq` order — duplicates with the same payload are silently absorbed (idempotent), duplicates with a *different* payload raise `InconsistentChunk` so a flapping upstream replica cannot silently corrupt the stream, and `is_final=True` cannot move once set. `state()` exposes a sorted-key snapshot (`next_expected_seq`, `buffered_seqs`, `gap_seqs`, `final_seq`, `is_complete`) so a heartbeat consumer can distinguish a *recovering* gap from a *stuck* one and decide when to give up. No I/O, no clocks, no retransmit logic — caller composes with their own transport policy. Stdlib-only Python. Two worked examples — 5-chunk stream delivered in arrival order `[0,2,4,1,2,3]` (note the duplicate seq=2) reassembles to `'The quick brown fox.'` exactly once with `delivered_count=5`, and a stream with a persistent gap at `seq=1` correctly stays `is_complete=False` even after `is_final=True` arrives at `seq=3`, plus a contradictory-payload duplicate of `seq=2` is rejected loudly with `InconsistentChunk` while a same-payload duplicate is silently absorbed. Composes with `tool-call-retry-envelope` (transport retries can produce duplicate chunks; this makes them safe), `partial-failure-aggregator` (one reassembler per parallel stream, aggregator decides what to do when M of N report `is_complete=False` past deadline), and `structured-error-taxonomy` (`InconsistentChunk` classifies as `do_not_retry` — the upstream is contradicting itself). |
| [`templates/structured-output-repair-loop`](templates/structured-output-repair-loop/) | Bounded, stuck-detecting repair loop for LLM calls that must return structured output (JSON/YAML/strict prose). Validates each attempt, feeds the validator's specific error back as a structured hint, fingerprints errors to detect "same mistake twice," and exits early with `parsed`/`stuck`/`exhausted`/`expired` so the caller can degrade or hard-fail instead of burning tokens. Stdlib-only reference loop, mini-validator, error-fingerprint hash with array-index normalisation, repair-turn template that forces minimal edits, and three worked examples (typo field repaired in one turn, stuck markdown-fence loop bailing on attempt 2, exhausted loop handed off to caller-side fallback) all runnable against the deterministic mock model. The loop algorithm behind upgrading `agent-output-validation`'s `repair_once` policy to a real loop. |
| [`templates/agent-cost-budget-envelope`](templates/agent-cost-budget-envelope/) | Per-call / per-session / per-day cost cap envelope with graceful-degrade tiers and a global kill switch. Every model call goes through `check(policy, ledger, request) → Decision` returning `allow`, `allow_degraded(tier=…)`, or `deny(reason=…, headroom_usd=…)`; deterministic evaluation order means a per-call trip never silently masks a per-session trip. Stdlib-only reference engine + JSONL ledger + policy linter + degrade-explainer prompt. Two worked examples — per-call cap trips and the request runs at the cheap tier ($0.36 → $0.056), and a session cap trips on the sixth call with `on_trip=kill` (denies stay denied for the rest of the session, ledger is not appended). The "is this next call allowed at all?" companion to `token-budget-tracker`'s post-hoc reporting. |
| [`templates/tool-call-circuit-breaker`](templates/tool-call-circuit-breaker/) | Per-tool circuit breaker with a `closed → open → half_open → closed` state machine, deterministic clock-injected reference engine, and JSONL event-log replay. `decide(policy, state, now, tool)` returns `allow`, `allow_probe`, or `denied_open(cooldown_remaining_s=…)`; tripping requires both a failure-rate threshold and a `min_calls` floor so two failures in a row cannot block the breaker. Half-open allows a bounded number of probes (`probe_max_concurrent`) and only closes after `probe_required_successes` consecutive probe successes; any probe failure re-opens with cooldown reset. Two worked examples — a `min_calls=5, threshold=0.5` policy trips on the fifth call (4/5 failures) and denies subsequent calls with `cooldown_remaining_s`, and an open breaker recovers cleanly through two probe successes and resets the failure window. The runtime-control complement to `tool-call-retry-envelope` (per-call retryability) and `agent-cost-budget-envelope` (per-call affordability): retry says "this call can be re-attempted," budget says "you cannot afford this call," circuit breaker says "this tool itself is unhealthy right now." |
| [`templates/tool-permission-grant-envelope`](templates/tool-permission-grant-envelope/) | Pre-call authorization layer for agent tool use: declarative grants (agent + tool + scopes + max_calls + expires_at + per-arg `arg_allow`) and a pure decision engine that turns `(grants, usage, request)` into `allow` or `deny(reason=…)`. Deterministic precedence (`no_grant` → `tool_not_in_grant` → `revoked` → `expired` → `scope_not_granted` → `argument_not_allowed` → `call_quota_exhausted` → `ok`) with quota check **last** so a spammed un-granted tool always reports the *real* reason instead of the misleading "out of quota". Replayable from a JSONL allowed-call event log; engine never mutates state. Stdlib-only reference engine + JSON Schema (draft 2020-12) + CLI exiting 0/1/2. Two worked examples — `arg_allow` denies a write to `/etc` with `field=dir, value=/etc` while letting `/tmp/mission-42` through, plus a scope-escalation attempt surfaces `missing=["admin"]`; and a JSONL-replayed quota counter reaches 3/3 and is denied with `used=3, max=3`, while a later call past `expires_at` reports `expired` (not stale `call_quota_exhausted`) because expiry is checked first. The pre-flight authz companion to `tool-call-retry-envelope` / `agent-cost-budget-envelope` / `tool-call-circuit-breaker` — only calls that pass *all four* should reach the tool implementation. |
| [`templates/agent-checkpoint-resume`](templates/agent-checkpoint-resume/) | Append-only JSONL checkpoint format + deterministic resume engine for long-running agent missions. Two records per step (`step_begin` with `input_hash` + `tools_planned`, `step_end` with `output_hash` + `tools_called` + `exit_state`), fsynced on append; a step is committed iff both records pair on `step_id`. `plan_resume(records, expected_input_hashes)` returns one of four states (`fresh`, `resume`, `invalidated`, `complete`) with `next_step_index` and the offending `step_id` when applicable. `input_hash` is canonical-JSON SHA-256 over `{prompt, prior_outputs}`; on resume the planner recomputes expected hashes for each step and the engine refuses to silently fast-forward past a committed step whose hash drifted (prompt edit, model swap, earlier-output change). Stdlib-only engine + CLI exiting 0/1. Two worked examples — host dies after writing step 2's `step_begin` and the resume plan correctly returns `next_step_index=2` with `committed_count=2` keeping steps 0-1 intact, and a between-runs `system: v1 → v2-tightened` prompt edit trips `state="invalidated", invalidated_step_id="s0", reason="input_hash_drift"` at the *earliest* drifted step rather than burning step-1 work on a doomed mission. Composes with `agent-decision-log-format` (same `step_id`/`exit_state` enum), `prompt-fingerprinting` (drop the fingerprint into the hashed prompt blob), and `tool-call-retry-envelope` (mid-step crash means in-flight tool calls must be retry-safe under the envelope). |
| [`templates/partial-failure-aggregator`](templates/partial-failure-aggregator/) | Pure aggregator that collapses N independent fan-out tool-call results into one structured verdict (`all_ok` / `partial_ok` / `all_failed` / `quorum_ok` / `quorum_failed`) plus an actionable `advice` (`proceed` / `proceed_degraded` / `retry_failed_only` / `abort`) so orchestrator branching is uniform across missions. Strict input validation (missing/duplicate ids and bad statuses raise), deterministic `first_failure` (input order, not lexicographic), `skipped_counts_as` rule lets you treat preflight-denied calls as ok / fail / ignore for quorum math without hiding the original status in the per-id map. Stdlib-only Python; CLI exits 0/1/2 mapped to advice so it drops into a shell `case` as-is. Two worked examples — 5-way fan-out with 3 ok + 2 failed under `mode=all` yields `partial_ok` + `retry_failed_only` and exit 1 with the failed ids preserved for re-issue, and a 3-region `mode=quorum, quorum=2, skipped_counts_as=fail` batch correctly yields `quorum_failed` + `abort` exit 2 instead of letting a preflight-denied region silently pass quorum. The fan-out reducer companion to `tool-call-retry-envelope` (per-call retryability) and `agent-decision-log-format` (verdict + advice are one decision-log line). |
| [`templates/model-fallback-ladder`](templates/model-fallback-ladder/) | Pure planner that climbs down an ordered list of model rungs until one succeeds or the ladder is exhausted, with two distinct skip mechanisms recorded in the hop trace: **preflight token-budget skip** (`reason_class="preflight_too_long"`) when the prompt exceeds a rung's `max_input_tokens` saves the round-trip without burning it, and **reason-class skip** (`reason_class="skip_on_reason_class"`) when a rung's `skip_on_reason_classes` matches the previous rung's failure class avoids retrying e.g. `content_filter` on the same vendor's smaller model. Caller supplies `call_fn(rung, prompt) → outcome` so the planner does no I/O and is fully testable with a deterministic mock; preflight skips deliberately do NOT update `last_failure_class` so a never-tried rung can't trigger downstream skip rules. Stdlib-only Python; CLI exits 0 on `ok`, 1 on `exhausted`. Two worked examples — primary `rate_limited` + secondary `ok` returns `verdict=ok, winning_rung_id=secondary, rungs_tried=2`, and a `content_filter` trip on primary cascades into a same-vendor reason-class skip plus a 48k-token preflight skip on the 32k-cap tertiary so the ladder exhausts with `rungs_tried=1, rungs_skipped=2`, distinct `reason_class` values for each skipped hop letting cost reports attribute the exhaustion correctly. Orthogonal to `partial-failure-aggregator` (try-in-order vs fan-out reduce); composes with `tool-call-retry-envelope` (per-rung retryability inside `call_fn`) and `agent-cost-budget-envelope` (a budget gate inside `call_fn` returns `error reason_class=other` to climb to a cheaper rung). |
| [`templates/structured-error-taxonomy`](templates/structured-error-taxonomy/) | Pure classifier that turns a raw error record (`source`, `vendor_code`, `http_status`, `message`) into a canonical `(class, retryability, attribution)` triple drawn from a small stable enum so retry, circuit-breaker, fallback-ladder and cost layers all branch on the same `class` instead of fragile vendor-message substring matching. First-match-wins rule table with a deterministic catch-all (`unknown` / `do_not_retry` / `unknown`) — the *correct* default is to refuse to retry an error you don't understand, not to silently retry it. Stdlib-only Python; CLI exits 0 if every input matched a non-default rule, 1 if any fell through to `unknown` (so triage / CI can flag new vendor codes), 2 on malformed input. Two worked examples — six-record clean batch covering every major class (rate_limited, content_filter, tool_timeout, host_io, auth, context_length) all match named rules and exit 0, and a three-record batch where one unknown vendor code surfaces as `class=unknown, matched_rule=default` and the run exits 1 while the other two records still get correct verdicts. Composes with `model-fallback-ladder` (`class` → `skip_on_reason_classes`), `tool-call-retry-envelope` (`retryability` → `retry_class_hint`), `tool-call-circuit-breaker` (only `attribution=tool` failures count toward a tool's failure rate), and `agent-cost-budget-envelope` (`quota_exhausted` means degrade, don't climb). |
| [`templates/llm-output-trust-tiers`](templates/llm-output-trust-tiers/) | Pure tier router for LLM outputs. Given a structured evidence record (`schema_ok`, `repair_count`, `source_class`, `blast_radius`, `canary_passed`, optional `override_tier`), routes the output into one of four tiers — `auto_apply` / `shadow_apply` / `human_review` / `quarantine` — with three structural guarantees: hard fails (`schema_ok=false`, `repair_count > 3`) always quarantine regardless of override; demotions stack so independent risk signals can each push the output one rung toward quarantine; caller `override_tier` can only DEMOTE, never promote, so a buggy caller cannot upgrade `human_review` to `auto_apply` (promotion attempts are recorded as `override_ignored_would_promote:T`). Stdlib-only Python; CLI exits 0 (all auto/shadow), 1 (at least one human_review, no quarantine), 2 (at least one quarantine or malformed input) so it drops into a CI / cron `case` as-is. Two worked examples — three clean records (one with `source_fresh + repair_count:1` correctly stops at `shadow_apply` rather than escalating because blast is reversible) exit 0, and a five-record batch covers every escalation path: schema-invalid → quarantine, `repair_count=4` → quarantine on the over-threshold guard, `cached_known + reversible + repair_count=1 + canary_failed` → human_review, pristine record with caller `override_tier=human_review` honored as a demotion. Composes with `agent-output-validation` and `structured-output-repair-loop` (source of `schema_ok` / `repair_count`), `structured-error-taxonomy` (a `tool_bad_input` upstream forces `source_class=fresh` even on cache hit), `agent-trace-redaction-rules` (quarantined outputs are safe to export as redacted forensic data), and `agent-decision-log-format` (one log line per verdict for queryable audit). |

## How to use a template

1. Browse the catalog above and open the template directory.
2. Read its `README.md` end-to-end — every template includes a "When to use" / "When NOT to use" section.
3. Copy the directory contents into your own repo (or fork this repo).
4. Find the **Adapt this section** block in the template README and edit the listed variables for your project.
5. Run the worked example. Each template ships with one.
