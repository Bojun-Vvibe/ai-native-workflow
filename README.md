# AI-Native Workflow Templates

Opinionated, reusable templates and patterns for running AI coding agents at scale — focused on spec-kitty missions, multi-agent orchestration, prompt-cache discipline, and review-loop patterns. Each template is self-contained, documented, and ships with a runnable example you can copy into your own repo and adapt.

One hundred seventy-six templates, grouped by what they do.

### Orchestration patterns

| [`templates/agent-trace-span-orphan-detector`](templates/agent-trace-span-orphan-detector/) | Pure stdlib detector for structural anomalies in an agent execution trace expressed as a list of spans. Catches the four classes of bug that silently corrupt downstream analysis (Honeycomb / Tempo / Jaeger UIs all *render* a broken trace without warning, so the operator sees a tidy tree that is missing branches): `orphan` (a `parent_span_id` references a `span_id` not in the batch), `multiple_roots` (more than one `parent_span_id=None` inside the **same** `trace_id` — per-trace_id scoping is the design choice, since a batch routinely spans multiple traces and "two roots in one batch" is normal but "two roots inside one trace_id" is the bug), `cycle` (parent chain hits the same span twice; walk bounded by `len(spans)` with a redundant safety net so a pathological input cannot push the detector into an infinite loop), and `cross_trace` (a span's parent lives in a *different* `trace_id`). Plus a soft `dangling_open` warning (no `finished_at` and not the most recent activity) so a forgotten `span.end()` surfaces before it inflates parent durations forever. Duplicate `span_id` raises `TraceValidationError` eagerly — the rest of the analysis would be ambiguous with two spans claiming the same id, and the *correct* default is to refuse to analyze. Findings sorted `(kind, span_id)` so two runs over the same input produce byte-identical output (cron-friendly diffing). Pure function over an in-memory list — no I/O, no clocks, no transport. Stdlib-only Python. Worked example: five synthetic traces (one healthy plus one per finding class — orphan, multiple_roots, cycle, cross+dangling) prove the per-trace_id root rule (case 05 has `trace_id="T-OTHER"` for the second `parent=None` span and is correctly *not* flagged as a second root inside `T5`), the cycle detection (parent chain `tool1↔tool2` flagged on both spans), and the dangling-open warning (an open span at `t=120` is flagged because `tool3` started later at `t=199`). Composes with `agent-trace-redaction-rules` (run this after redaction to confirm structure didn't drift), `agent-decision-log-format` (one log line per finding sharing `span_id`), and `structured-error-taxonomy` (orphan/cycle/multiple_roots → `attribution=tool` instrumentation bug; cross_trace → `attribution=user` caller mixing batches; dangling_open → `do_not_retry`, the trace is already on disk). |

| [`templates/agent-handoff-message-validator`](templates/agent-handoff-message-validator/) | Payload-level validator for the JSON one agent passes to another at a handoff boundary (scout → actor, planner → implementer, implementer → reviewer). Distinct from `agent-handoff-protocol` (transport envelope `done`/`partial`/`unrecoverable`): this template validates the *payload inside* — required fields + types, `from_agent != to_agent`, `task_id` regex, `next_action` enum, soft/hard summary char caps, per-question length caps, artifact `{kind, ref}` shape with duplicate-ref warning, internal `artifact:<ref>` mentions in summary that must resolve, and an optional `banned_tokens` substring scan over every stringy field. Returns a `ValidationResult(ok, errors, warnings)` and never raises on bad input — the gate runs at every handoff and a malformed message must produce a structured failure, not a stack trace. Stdlib-only. Worked example covers four scenarios — clean handoff (PASS), missing-fields + short `task_id` (3 structural errors), bad-refs + `from_agent==to_agent` + bad enum + duplicate ref (5 errors + 1 warning), and a banned-token leak in the summary caught by the optional blocklist. Composes with `agent-handoff-protocol` (transport vs payload), `agent-output-validation` (validates downstream JSON outputs; this validates the upstream handoff message), and `agent-trace-redaction-rules` (scrub before persistence). |
| [`templates/agent-tool-call-loop-detector`](templates/agent-tool-call-loop-detector/) | Pre-dispatch detector that fires when an autonomous agent is stuck in a degenerate tool-call loop, so the host can break out, escalate, or inject a corrective system message before burning more budget on the same call. Three independent signals over a configurable tail window (default 8): `exact_repeat` (same `(tool, canonical_args)` ≥ `repeat_threshold` times — default 3), `abab_cycle` (alternating two-call pattern of length ≥ `cycle_min_len` — default 4), and `no_progress` (the entire window collapses to a single distinct call). Args are canonicalized via sorted-key JSON so `{a:1,b:2}` and `{b:2,a:1}` hash equal — the naive "same dict twice" check would miss agent re-issues that only differ in key ordering. Returns a structured `LoopReport(looped, reason, detail)` so the host can log, emit a metric, and decide on escalation. Pure stdlib. Worked example: 5 synthetic histories (1 healthy + 4 pathological — exact-repeat grep loop, ABAB read/edit cycle, no-progress single call only, key-reordered http_get) — 4 loops correctly flagged with their canonical-args fingerprint, the healthy progression correctly passes through. The *semantic* signal — "you are repeating yourself" — that fires long before `agent-loop-iteration-cap` (hard wall on total iterations) and `agent-step-budget-monitor` (cost / latency budgets) hit their walls. |

| [`templates/tool-call-deduplication`](templates/tool-call-deduplication/) | In-process, signature-hash deduplication of tool calls — the sibling of `tool-call-retry-envelope` for the case where the *agent itself* re-issues the same logical call within seconds (thought loops, partial-trace confusion). Caller mints no key; the host hashes `(tool_name, canonical(identity_args))` with a closed `identity_fields` allowlist so volatile metadata (request-ids, timestamps) does not defeat dedup. Floats in identity-args raise `CanonicalizationError` (silent precision loss is a correctness trap); dict-key order does not change the hash; lazy expiry on access keeps memory bounded. Stdlib-only `DedupCache(window_seconds, now_fn)` with deterministic injected clock. Two worked examples — agent loops and a 50ms-later re-issue of `read_file` is served from cache with `original_call_id` preserved for the audit link, and an `identity_fields=["query","limit"]` allowlist correctly absorbs two calls that differ only in `request_id`/`now` while `limit=25→50` correctly re-executes, plus an 11s clock advance evicts past the 10s window and a literal `0.875` in identity-args is rejected with a JSON pointer to the offending field. | [`templates/tool-result-cache`](templates/tool-result-cache/) | Content-addressed result cache for **deterministic** tool calls — the long-lived, positive sibling of `tool-call-deduplication`. Caller declares `safe_tools` (`frozenset`) at construction; `write` for any tool outside the allowlist raises `UnsafeCacheError` and increments `state.refused_unsafe_writes` so a `read_clock` / `web_fetch` / `list_files` cannot be cached by accident. Per-entry `ttl_s` (not a single shared window), lazy expiry on lookup, never read-extends-TTL (a cache that refreshes on read is a memory leak in disguise). `cache_key(tool, args, identity_fields=…)` reuses the same canonicalizer as the dedup template (sorted keys, no whitespace, floats raise `CacheKeyError` with a JSON pointer). Stdlib-only `ToolResultCache(safe_tools, now_fn)` with injected clock. Three-part worked example — `sha256_of_file` HIT across two calls with different dict-key order and different `request_id` carries `source_call_id="call-A"` for the audit link; an `read_clock` write attempt is refused loudly (`refused_unsafe_writes=1`); a `read_file` entry written with `ttl_s=5.0` HITs at `t=1004.95` and is evicted at `t=1005.55` (`evictions: 0 → 1`). Composes with `tool-call-deduplication` (run dedup first, this cache second), `structured-error-taxonomy` (caller must classify success before `write` — failures are never cached), and `agent-decision-log-format` (log every HIT with `source_call_id`). |
| [`templates/streaming-chunk-reassembler`](templates/streaming-chunk-reassembler/) | Pure, single-stream reassembler for streamed LLM/tool chunks delivered out of order, duplicated, or with persistent gaps. `accept(chunk) -> list[chunk]` returns only the chunks that became deliverable in strict `seq` order — duplicates with the same payload are silently absorbed (idempotent), duplicates with a *different* payload raise `InconsistentChunk` so a flapping upstream replica cannot silently corrupt the stream, and `is_final=True` cannot move once set. `state()` exposes a sorted-key snapshot (`next_expected_seq`, `buffered_seqs`, `gap_seqs`, `final_seq`, `is_complete`) so a heartbeat consumer can distinguish a *recovering* gap from a *stuck* one and decide when to give up. No I/O, no clocks, no retransmit logic — caller composes with their own transport policy. Stdlib-only Python. Two worked examples — 5-chunk stream delivered in arrival order `[0,2,4,1,2,3]` (note the duplicate seq=2) reassembles to `'The quick brown fox.'` exactly once with `delivered_count=5`, and a stream with a persistent gap at `seq=1` correctly stays `is_complete=False` even after `is_final=True` arrives at `seq=3`, plus a contradictory-payload duplicate of `seq=2` is rejected loudly with `InconsistentChunk` while a same-payload duplicate is silently absorbed. Composes with `tool-call-retry-envelope` (transport retries can produce duplicate chunks; this makes them safe), `partial-failure-aggregator` (one reassembler per parallel stream, aggregator decides what to do when M of N report `is_complete=False` past deadline), and `structured-error-taxonomy` (`InconsistentChunk` classifies as `do_not_retry` — the upstream is contradicting itself). |

Worked example: 8 cases — clean tagged Python block, missing tag, inconsistent `py`+`python` in same doc, content mismatch (`python` body is JSON), unclosed `bash` fence, unknown-tag with allowlist, non-lowercase tag, and the tilde-fence-with-inner-backticks edge case. | [`templates/agent-tool-call-timestamp-monotonicity-validator`](templates/agent-tool-call-timestamp-monotonicity-validator/) | Pure stdlib validator that scans an agent's tool-call trace for **timestamp** anomalies that silently corrupt downstream analysis — the temporal counterpart to `agent-trace-span-orphan-detector`'s structural checks. The failure mode it catches: a trace is replayed, merged across worker threads, or stitched together from multiple log shards, and the timestamps end up out of order. Most analytics tools (latency histograms, span trees, retry-rate dashboards) *render* such a trace without warning — the bug is invisible until someone notices that "this tool call apparently took -8 seconds" or "this agent session apparently lasted 47 minutes between two adjacent calls, even though the host was online the whole time." Five findings: `non_monotonic` (within a single thread, two consecutive calls have `start_ms[i] < start_ms[i-1]` — classic out-of-order replay / shard merge bug), `duplicate_timestamp` (two consecutive calls share the exact same `start_ms` — log-line collision; ordering now ambiguous), `clock_jump_forward` (positive gap larger than `max_gap_ms` default 60s — NTP step, laptop sleep / resume, or a missing intermediate event), `negative_duration` (`end_ms < start_ms` — always a bug, impossible in real time), `future_timestamp` (`end_ms > now_ms` — worker clock drifted ahead of the orchestrator). **Per-thread grouping is the load-bearing design choice**: a multi-threaded agent *will* legitimately interleave timestamps in the global stream — that's the whole point of running multiple threads — and flagging cross-thread interleaving as out-of-order would produce a deluge of false positives, so the validator only complains when ordering breaks *within a single thread*, which is the only case that's actually a bug (case 02 in the worked example proves this — two threads with globally-interleaved starts pass cleanly). `now_ms` is optional so the same validator runs offline (post-mortem, batch eval — pass `None` and `future_timestamp` simply doesn't fire) or live (pass the orchestrator's current timestamp). `max_gap_ms=60_000` is long enough that a slow `bash` build doesn't false-fire and short enough that a 5-minute laptop-sleep is caught immediately. Eager refusal on bad input — missing keys / wrong types raise `TimestampValidationError` rather than letting a malformed trace silently audit clean. Pure function, stdlib only. Findings sorted `(kind, thread, idx, detail)` for byte-identical re-runs. Worked example: eight cases — single-thread clean, interleaved-threads clean (proves per-thread grouping), non-monotonic, duplicate-timestamp, 90-second clock-jump, negative-duration, future-timestamp, and a combined case that fires three findings on one thread. Five finding kinds, sorted by `(offset, kind)`: `undefined_first_use` (acronym appears for the first time with no preceding expansion — fired on every undefined acronym including single uses; the detail string carries the acronym and the sentence so the repair prompt is one interpolation away), `never_expanded` (acronym appears at least `min_repeats` times — default 2 — and is never expanded anywhere; distinct severity from `undefined_first_use` because a one-off undefined acronym is often acceptable but a repeatedly-used one without a binding is almost always wrong; both findings fire on the same offset for repeated-and-undefined cases so the reviewer sees both severities in adjacent report lines), `inconsistent_expansion` (same acronym, multiple expansions with different long forms — `LLM (large language model)` and later `LLM (language learning module)` — almost always a model artifact where the generator lost the binding across paragraphs; the detail string carries both forms verbatim so the reviewer can decide which is correct or whether the model meant two distinct concepts and chose a colliding acronym), `redundant_re_expansion` (same acronym re-expanded with the same long form — soft warning kind; readers do not need the second `service-level objective (SLO)` once the binding is established; demote to info-level if your house style allows re-expansion at section boundaries), and `lowercase_after_acronym` (a lowercased form of an acronym introduced uppercase appears later as a standalone word — often a model artifact where the model briefly forgets the term is an acronym and writes it as an ordinary noun). An acronym candidate is a token of `>= min_len` (default 2) ASCII uppercase letters with optional digit suffix (`SLO`, `HTTP2`, `TLS13`); a configurable `allowlist` (default: a small set of universally-known software-engineering acronyms — `OK`, `URL`, `API`, `JSON`, `HTTP`, `HTTPS`, `CSS`, `HTML`, `SQL`, `XML`, `YAML`, `CSV`, `PDF`, `RAM`, `CPU`, `GPU`, `OS`, `UI`, `UX`, `ID`, `PR`, `CI`, `CD`, `IO`, `AM`, `PM`, `UTC`, `TCP`, `UDP`, `DNS`, `TLS`, `SSL`, `SSH`, `IP`, `MAC`, `USB`, `SDK`, `IDE`, `REST`, `RPC`, `TODO`, `FAQ`, ...) is never flagged; `extra_known` lets the caller pre-declare project-specific acronyms (the team knows what `RPO` means). Inside fenced code blocks, inline code spans, and Markdown link URLs (the `(...)` half of `[text](url)`), tokens are skipped — URLs frequently contain uppercase tokens that are not English acronyms (env vars, HTTP header names) and would false-positive heavily. Pure function over `str` — no Markdown parser, no language detection, no networking; single forward pass with one O(N) re-scan to compute the cross-document expansion table. Stdlib-only Python. Worked example: 6 cases — clean doc with both `SLO` and `RPO` expanded at first use passes silently; `SLO` used at offset 4 with no expansion fires single `undefined_first_use`; `RPO` used three times never expanded fires both `never_expanded` and `undefined_first_use` at the same offset; `LLM` expanded twice with different long forms fires `inconsistent_expansion` carrying both verbatim; `SLO` legitimately defined paragraph 1 then re-defined paragraph 3 fires the soft `redundant_re_expansion`; and a case proving both `lowercase_after_acronym` (`slo` flagged because `SLO` was introduced earlier) and the allowlist behavior (`API` correctly NOT flagged despite no expansion). Composes with `agent-output-validation` (the canonical case — the repair prompt asks the model to insert `<long form> (<acronym>)` at the offending sentence), `structured-output-repair-loop` (the `(offset, kind, acronym)` tuple is a stable fingerprint — same tuple twice in a row → bail), `llm-output-quote-style-consistency-validator` and `llm-output-emphasis-marker-consistency-validator` and `llm-output-bullet-terminal-punctuation-consistency-validator` (orthogonal typographic / structural validators on the same blob; same fence-awareness convention, same deterministic sort, one CI step can union them), and `structured-error-taxonomy` (`do_not_retry / attribution=model` for the four binding-loss kinds; `redundant_re_expansion` is `info` and never fails CI). |

Fence parser is CommonMark-aligned; inline-code stripping uses run-length-balanced backtick matching. Pure function over `str` — no I/O, no networking, no normalization. Stdlib-only Python. Findings sorted `(offset, kind)` so byte-identical re-runs make diff-on-the-output a valid CI signal. Configurable `allowlist` of codepoints (e.g. tolerate `U+200D` in projects that ship emoji ZWJ sequences) and `suppress_in_code=True` for docs that legitimately demonstrate invisibles. | [`templates/llm-output-json-trailing-comma-and-comment-detector`](templates/llm-output-json-trailing-comma-and-comment-detector/) | Pure stdlib detector for JSONC artifacts that leak into output that is supposed to be strict JSON: trailing commas before `]` / `}`, and `//` line / `/* … */` block comments. The single most common reason a model's "JSON" reply fails `json.loads` despite parsing fine in a JSONC-tolerant linter (VS Code's settings parser, JSON5, hjson). It deserves its own focused detector rather than a generic "json invalid" branch because (1) the **fix is mechanical** — strip the comments, trim the comma — and a one-shot repair prompt with the exact offset is much cheaper than asking the model to re-emit the whole document; (2) **distinguishing JSONC drift from real schema violations** prevents the repair loop from chasing the wrong bug — if `repair_count` keeps firing on `,}` patterns the right fix is system-prompt tuning ("strict JSON only"), not harder schema validation; (3) the **lexer is small** (~150 lines, one pass, string-aware) and a real JSON parser will reject the whole document and tell you nothing useful about *which* of the four artifacts is present where. Five finding kinds: `trailing_comma_object` (`,}`), `trailing_comma_array` (`,]`), `line_comment` (`// …` to end of line), `block_comment` (`/* … */`), and `unterminated_block_comment` (`/*` with no closing — partial / torn output, the validator REFUSES to auto-repair this case because the correct action is to re-emit, not guess where `*/` was meant to go). Includes opt-in `strip_artifacts(text)` that performs a deterministic mechanical repair (no model call) returning `(cleaned_text, findings)`; refuses to touch input containing an unterminated block comment. The lexer is **string-aware** — a comma and `/* */` lookalike inside a JSON string value (`"SELECT a, /* not a real comment */ b WHERE x = 3,"`) produces ZERO findings (case 07 proves it; this is the test the naïve `text.replace(",}", "}")` approach always fails, and why this template ships a real lexer). Backslash-escaped quotes inside strings are honored (`\\` and `\"`). Each `Finding` carries `offset`, `line_no`, `column`, `kind`, and a short literal `snippet` (newlines / tabs escaped) so the report is one line per finding and pipes cleanly through grep/awk; sorted by `offset` for byte-identical re-runs. Pure function over `str` — no I/O. Stdlib-only Python. Worked example: 8 cases — strict-clean JSON passes silently, `,}` and `,]` flagged at the comma offset (not the bracket) so a one-line patch is `text[:off] + text[off+1:]` and the cleaned text round-trips through `json.loads` with the expected keys (proof in the captured output), `// header` and `/* removed for v2 */` reported with full delimiter snippets and stripped cleanly, the **kitchen-sink case** with all four kinds in one document reports five findings in offset order and the cleaned output is one `json.loads` away from a usable config dict, the in-string false-positive negative test produces zero findings (the validator's most important invariant), and the unterminated-block-comment case is reported but `strip_artifacts` deliberately refuses (no `--- after strip_artifacts ---` block in the output, proving the refusal). Composes with `structured-output-repair-loop` (a `repair_count == 1` failure whose findings are entirely from this detector should be rewritten via `strip_artifacts` rather than re-prompted — byte-deterministic, doesn't count against the model-repair budget), `agent-output-validation` (feed `(kind, line_no, column)` into the schema-repair prompt when the model has to re-emit), `partial-json-tail-recovery` (runs *before* this detector when upstream reports the stream was cut — recovery presupposes truncation, this detector presupposes a complete document; wired in that order, a torn JSONC document still produces a usable parse), `structured-error-taxonomy` (`trailing_comma_*` and `*_comment` → `retryable_after_repair / attribution=model`; `unterminated_block_comment` → `retryable_after_repair / attribution=upstream`), and `agent-decision-log-format` (log `repair_kind=jsonc_strip, artifacts=N, model_call=false` so cost reports attribute mechanical repairs separately from re-prompts). |

## How to use a template

1. Browse the catalog above and open the template directory.
2. Read its `README.md` end-to-end — every template includes a "When to use" / "When NOT to use" section.
3. Copy the directory contents into your own repo (or fork this repo).
4. Find the **Adapt this section** block in the template README and edit the listed variables for your project.
5. Run the worked example. Each template ships with one.
