# AI-Native Workflow Templates

Opinionated, reusable templates and patterns for running AI coding agents at scale — focused on spec-kitty missions, multi-agent orchestration, prompt-cache discipline, and review-loop patterns. Each template is self-contained, documented, and ships with a runnable example you can copy into your own repo and adapt.

## Catalog

One hundred templates, grouped by what they do.

| [`templates/tool-call-deduplication`](templates/tool-call-deduplication/) | In-process, signature-hash deduplication of tool calls — the sibling of `tool-call-retry-envelope` for the case where the *agent itself* re-issues the same logical call within seconds (thought loops, partial-trace confusion). Floats in identity-args raise `CanonicalizationError` (silent precision loss is a correctness trap); dict-key order does not change the hash; lazy expiry on access keeps memory bounded. Composes with `tool-call-retry-envelope` (cross-process, caller-keyed twin), `agent-decision-log-format` (log every `verdict=use_cached` with `original_call_id`), and `tool-permission-grant-envelope` (dedup runs *after* the grant decision so denied calls are never cached). |
| [`templates/tool-result-cache`](templates/tool-result-cache/) | Content-addressed result cache for **deterministic** tool calls — the long-lived, positive sibling of `tool-call-deduplication`. Per-entry `ttl_s` (not a single shared window), lazy expiry on lookup, never read-extends-TTL (a cache that refreshes on read is a memory leak in disguise). `cache_key(tool, args, identity_fields=…)` reuses the same canonicalizer as the dedup template (sorted keys, no whitespace, floats raise `CacheKeyError` with a JSON pointer). Stdlib-only `ToolResultCache(safe_tools, now_fn)` with injected clock. Three-part worked example — `sha256_of_file` HIT across two calls with different dict-key order and different `request_id` carries `source_call_id="call-A"` for the audit link; an `read_clock` write attempt is refused loudly (`refused_unsafe_writes=1`); a `read_file` entry written with `ttl_s=5.0` HITs at `t=1004.95` and is evicted at `t=1005.55` (`evictions: 0 → 1`). Composes with `tool-call-deduplication` (run dedup first, this cache second), `structured-error-taxonomy` (caller must classify success before `write` — failures are never cached), and `agent-decision-log-format` (log every HIT with `source_call_id`). |
| [`templates/streaming-chunk-reassembler`](templates/streaming-chunk-reassembler/) | Pure, single-stream reassembler for streamed LLM/tool chunks delivered out of order, duplicated, or with persistent gaps. `accept(chunk) -> list[chunk]` returns only the chunks that became deliverable in strict `seq` order — duplicates with the same payload are silently absorbed (idempotent), duplicates with a *different* payload raise `InconsistentChunk` so a flapping upstream replica cannot silently corrupt the stream, and `is_final=True` cannot move once set. `state()` exposes a sorted-key snapshot (`next_expected_seq`, `buffered_seqs`, `gap_seqs`, `final_seq`, `is_complete`) so a heartbeat consumer can distinguish a *recovering* gap from a *stuck* one and decide when to give up. No I/O, no clocks, no retransmit logic — caller composes with their own transport policy. Stdlib-only Python. Two worked examples — 5-chunk stream delivered in arrival order `[0,2,4,1,2,3]` (note the duplicate seq=2) reassembles to `'The quick brown fox.'` exactly once with `delivered_count=5`, and a stream with a persistent gap at `seq=1` correctly stays `is_complete=False` even after `is_final=True` arrives at `seq=3`, plus a contradictory-payload duplicate of `seq=2` is rejected loudly with `InconsistentChunk` while a same-payload duplicate is silently absorbed. Composes with `tool-call-retry-envelope` (transport retries can produce duplicate chunks; this makes them safe), `partial-failure-aggregator` (one reassembler per parallel stream, aggregator decides what to do when M of N report `is_complete=False` past deadline), and `structured-error-taxonomy` (`InconsistentChunk` classifies as `do_not_retry` — the upstream is contradicting itself). |

| [`templates/tool-call-argument-schema-coercer`](templates/tool-call-argument-schema-coercer/) | Coercion layer between the model's emitted tool-call argument dict and the host's actual tool dispatch. | [`templates/partial-output-checkpointer`](templates/partial-output-checkpointer/) | Periodic byte-level checkpoint of streaming output so a SIGTERM mid-flight leaves recoverable state — the byte-stream sibling of `agent-checkpoint-resume`. `Checkpointer(stream_id, FlushPolicy(every_bytes, every_seconds), sink_write, log_write, now_fn)` flushes when **either** size or time threshold trips (empty buffer never flushes, so a quiet stream produces no spurious records). Each JSONL record carries `bytes_committed` and a `running_sha256` over **all** bytes written so far, so `recover(log_text, expected_stream_id)` returns a `RecoveryPlan(bytes_committed, last_running_sha256, intact_records, torn_trailing_record)` the caller seeks to and verifies in one pass. Torn-write handling is first-class and asymmetric: a parse failure on the **trailing** line is reported as `torn_trailing_record=True` and the resume point comes from the previous good record; a parse failure on **any other** line raises `CheckpointError` because silent middle-skip would hide real disk damage. Sequence numbers must be dense from 0; a gap raises. Stdlib-only; sink and log writers are injected so the engine itself does no I/O. Five-part worked example — 1200 B stream under `every_bytes=400` produces exactly 2 policy-flushes + 1 finalize-flush, the running SHA matches a direct hash of the sink, a truncated trailing line recovers cleanly to `bytes_committed=960` and verifies on disk, a corrupt non-trailing record raises loudly, and append-after-finalize raises. Composes with `agent-checkpoint-resume` (step-level vs byte-level), `streaming-chunk-reassembler` (its in-order output feeds `append`), and `structured-error-taxonomy` (torn-trailing → `host_io / retryable_after_recover`; non-trailing torn → `do_not_retry / attribution=host`). |

| [`templates/model-fallback-ladder`](templates/model-fallback-ladder/) | Pure planner that climbs down an ordered list of model rungs until one succeeds or the ladder is exhausted, with two distinct skip mechanisms recorded in the hop trace: **preflight token-budget skip** (`reason_class="preflight_too_long"`) when the prompt exceeds a rung's `max_input_tokens` saves the round-trip without burning it, and **reason-class skip** (`reason_class="skip_on_reason_class"`) when a rung's `skip_on_reason_classes` matches the previous rung's failure class avoids retrying e.g. `content_filter` on the same vendor's smaller model. | [`templates/weighted-model-router`](templates/weighted-model-router/) | Deterministic, weight-respecting router for picking one of N model backends per request — the *probabilistic* sibling of `model-fallback-ladder`'s sequential climb. Selection rule is weighted-rendezvous (HRW) hashing: `score(b) = b.weight / -ln(uniform_hash(route_key, b.name))`, pick max with lexicographic tiebreak on `backend.name`. Two correctness properties the naïve `random.choices(weights=…)` implementation lacks: (1) **deterministic given a route_key** — same `route_key` lands on the same backend across processes / restarts / replays, so pinning user X to model A for an A/B actually works and a recorded trace replays against the original backend; (2) **stable under weight edits** — bumping `model-B` from `weight=20→25` only reroutes ~3.88% of keys (worked example) instead of the ~67% a hash-mod router would reshuffle; only keys whose top-2 score straddled the changed bucket move. `Backend(weight=…)` raises `InvalidWeight` on `<= 0` or non-finite (silent zero-weight is a typo trap); duplicate names raise; `route(key, exclude={…})` filters drained backends *before* selection so a circuit-open backend can never win and `NoEligibleBackend` raises loudly when every candidate is excluded rather than degrading to random. Stdlib-only (`hashlib.blake2b`, `math`); pure function — no I/O, no clocks. Worked example proves all four invariants: 70/20/10 weights produce 70.16/19.54/10.30 over 10 000 keys, the same key returns the same backend on every call, an `exclude={"model-A"}` reroutes only the keys that were on `model-A`, and excluding all three raises `NoEligibleBackend`. Composes with `model-fallback-ladder` (router picks the first backend; ladder governs what to do on failure), `tool-call-circuit-breaker` (breaker passes the unhealthy backend's name into `exclude=`), `tool-call-replay-log` (recorded `route_key` reproduces the original upstream on replay), and `agent-cost-budget-envelope` (derive `weight` from `remaining_budget / per_call_cost` to drain expensive backends as their dollar cap is consumed). |

Thread-safe (`threading.Lock`-guarded refill + consume), monotonic-clock based, with a `RetryStats` accumulator (`attempts`, `retries_used`, `retries_denied`, `successes`, `failures`) for observability. Worked example: 8 concurrent flaky callers (60% transient failure) sharing one 5-token budget — 3 succeed outright, 1 burns its per-call cap, 4 are *denied* retries because earlier callers drained the bucket; runtime asserts the invariant `retries_used <= capacity (5)` so the budget never leaks. The "stop the cascade" companion to `tool-call-retry-envelope`'s per-retry safety contract and `tool-call-circuit-breaker`'s tripping logic — retry envelope makes a single retry safe, this template caps how many retries the whole process is allowed to fire during a brownout, and the circuit breaker decides when to stop trying entirely. |
| [`templates/streaming-tool-call-buffer`](templates/streaming-tool-call-buffer/) | Per-`index` buffer for streaming tool-call deltas (OpenAI-compatible / Anthropic SSE) that emits each call exactly once when its arguments are complete and parse as a JSON object. Detects completion via three independent signals: explicit `finalize` flag, the stream advancing to a higher index, or end-of-stream — so latency-sensitive calls dispatch as soon as the model moves on instead of waiting for the whole stream. Bad calls route to a separate `on_malformed` sink (with `index`, `name`, `raw_arguments`, parse error) instead of being silently dropped. Stdlib-only Python; worked example replays a 12-delta session producing 2 dispatched calls (one fragmented across 5 chunks, one finalize-only zero-arg) and 1 quarantined call (truncated mid-object). The "exactly once, validated" wire-side companion to `tool-call-retry-envelope` (idempotency across retries) and `llm-output-jsonschema-repair` (semantic repair of the parsed args). |
| [`templates/llm-output-jsonschema-repair`](templates/llm-output-jsonschema-repair/) | Local-only repair pass for LLM outputs that are supposed to match a JSON Schema but don't quite — applied **before** spending a model round-trip asking for a fix. Strips Markdown code fences and conversational preambles ("Sure, here is…"), normalizes smart quotes, drops trailing commas, injects schema-declared defaults for missing required fields, drops `additionalProperties: false` violations, and losslessly coerces types (`"42" → 42`, `"false" → False`). Returns a `RepairResult` with the parsed value, the ordered list of repairs applied, and any remaining violations — so the caller decides whether to accept, escalate to a model retry, or quarantine. Stdlib-only (no `jsonschema` dep), supports the practical subset (`type`, `properties`, `required`, `default`, `enum`, `items`, `additionalProperties`). Worked example feeds five fixtures through one pass: 4 land cleanly with zero round-trips (fenced+preamble, smart quotes + trailing commas, missing-required-with-default + extra property, type-mismatch coercion), and 1 (missing required field with no default) is correctly quarantined for the caller to escalate. The "fix it locally first" companion to `tool-call-result-validator` and `streaming-tool-call-buffer`. |
| [`templates/llm-output-fence-extractor`](templates/llm-output-fence-extractor/) | Robust extractor that pulls fenced code blocks out of LLM markdown output and survives the messes models actually produce: nested fences (a 4-backtick wrapper containing inner 3-backtick blocks returns one outer block with the inner fences preserved verbatim), tilde fences (`~~~`), language tags decorated with attributes (`python {.numberLines}`) or trailing punctuation (`python:`, `python,`) all normalize to `python`, and **truncated final blocks** the model never closed are returned with `terminated=False` so callers can repair / re-prompt / discard instead of silently shipping a half-snippet (the failure mode that justifies the template — naive regex returns nothing for this input). Returns a structured `CodeBlock(lang, info, body, fence_char, fence_len, indent, start_line, end_line, terminated)` with both normalized and raw info strings preserved. Up to 3 leading spaces of indent on the opening fence are tolerated (CommonMark behavior); closing fence must be the same char and `>=` the opening length. Stdlib-only Python (~110 lines, regex only). Worked example: six real LLM-output shapes — plain `python` block, 4-backtick wrapper containing inner 3-backtick `python`, `~~~ruby`, no-language block, `python {.numberLines startFrom=1}` + `python:` both normalize to `python`, and a truncated final `def slow():` block returned with `terminated=False` and the partial body preserved. Composes with `partial-json-streaming-parser` (extract a fenced JSON block then stream-parse), `structured-output-repair-loop` (a `terminated=False` block is the trigger to feed the partial back for continuation), and `llm-output-jsonschema-repair` (extract first, then schema-repair the body). |

| [`templates/cross-tick-state-handoff`](templates/cross-tick-state-handoff/) | File-locked, atomic JSON state envelope for handing partial work between successive ticks of an autonomous dispatcher (or any cron-like loop). `HandoffStore(path, schema_version)` wraps `fcntl.flock` on a sibling `.lock` file, `tempfile.mkstemp` + `os.replace` for atomic commits, and a top-level `_schema` field the reader checks before unwrapping. `with store.transaction() as state: ...` yields a mutable dict pre-loaded with the committed envelope and atomically commits on clean exit; on any exception the temp file is unlinked and the previous envelope is left intact. Worked example simulates four successive dispatcher ticks, then proves crash-safety by raising inside `transaction()` and confirming the envelope is unchanged, then proves the schema guard by trying to load with `schema_version=2` against an envelope written at version 1. | [`templates/model-output-truncation-detector`](templates/model-output-truncation-detector/) | Heuristic detector for "the model hit `max_tokens` mid-thought" plus a continuation-prompt builder. `detect(text, finish_reason=None)` fuses the upstream `finish_reason` signal (when available) with structural anomalies in the text — unclosed code fences, unbalanced brackets (string-literal aware), ends-mid-word (sentence-terminator window), ends-mid-bullet — and returns a four-level verdict (`CLEAN`/`SUSPICIOUS`/`LIKELY_TRUNCATED`/`TRUNCATED`). Asymmetric trust: `finish_reason="length"` is authoritative regardless of how clean the tail looks; `finish_reason="stop"` *downgrades* structural signals because models legitimately produce odd-looking endings. `build_continuation_prompt(verdict, original_request, partial)` returns a follow-up prompt that bundles the detected signals, the original request verbatim, and the exact last 200 chars produced, with an explicit "do NOT restate" instruction to avoid the stitch-seam failure mode. Stdlib-only, deterministic, O(len(text)). Worked example exercises six representative cases (clean stop, explicit length, unclosed code fence, mid-bullet, innocent stop with code block, mid-word) and prints the continuation prompt for the worst case. |

## How to use a template

1. Browse the catalog above and open the template directory.
2. Read its `README.md` end-to-end — every template includes a "When to use" / "When NOT to use" section.
3. Copy the directory contents into your own repo (or fork this repo).
4. Find the **Adapt this section** block in the template README and edit the listed variables for your project.
5. Run the worked example. Each template ships with one.
