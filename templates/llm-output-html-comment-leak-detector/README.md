# `llm-output-html-comment-leak-detector`

Pure-stdlib detector for leaked HTML comments (`<!-- ... -->`) in an
LLM Markdown / prose output blob.

LLMs are trained on a lot of HTML and Markdown that uses HTML
comments for editorial scaffolding: TOC anchors, "TODO" notes,
prompt-engineering snippets, system reminders, autogenerated markers.
When asked to produce a Markdown deliverable, a model occasionally
leaks one of these comments into the output. The comment is INVISIBLE
in the rendered Markdown (GitHub, most wikis, most chat preview cards)
but survives forever in the source — including in `git log`, `gh pr
view`, and any diff.

Three finding kinds:

- `html_comment` — a complete `<!-- ... -->` comment. Reported per
  occurrence with the 1-based start line and the inner text
  (truncated to 80 chars). The full comment text is in the report
  so a reviewer does not need to jump back to the source.
- `unterminated_comment` — `<!--` opened but never closed before
  end of blob. Usually means the model truncated mid-comment OR
  emitted a literal `<!--` in prose by mistake. Reported once with
  the start line.
- `suspicious_marker` — the inner text matches one of a tiny
  fixed set of canonical "this is editorial scaffolding, not user
  content" phrases: `todo`, `fixme`, `xxx`, `prompt:`, `system:`,
  `do not ship`, `internal`, `note to self`. Reported in ADDITION
  to the `html_comment` finding for the same comment, so the report
  shows both the byte-level and the semantic concern. Matching is
  case-insensitive.

## When to use

- Pre-publish gate on any LLM-generated **commit message body**,
  **PR description**, or **issue body** before `gh` / `git` writes
  it to a permanent record. A leaked `<!-- prompt: ... -->` in a
  PR description is a permanent prompt-leak in the org's git
  history.
- Pre-flight for an LLM-drafted **runbook** / **status update** /
  **changelog** before paste into a wiki: the comment is invisible
  in the rendered view but downstream consumers (release-notes
  scripts, RSS feeds, search indexers) will index its contents.
- Audit step in a **review-loop** that promotes
  [`agent-output-validation`](../agent-output-validation/)'s
  `repair_once` policy: a `suspicious_marker` finding feeds the
  offending comment text back into the repair prompt with a
  one-shot instruction ("delete the HTML comment on line N").
- Cron-friendly: findings are sorted by `(line_number, kind)` and
  the report is rendered in a deterministic format, so byte-
  identical output across runs makes diff-on-the-output a valid
  CI signal.

## Inputs / outputs

```
detect_html_comment_leaks(text: str) -> list[Finding]

Finding(kind: str, line_number: int, detail: str)
```

- `text` — the LLM output to scan. Must be `str` (raises
  `ValidationError` otherwise).
- `Finding.line_number` is 1-based and points at the line where
  `<!--` STARTS (not where `-->` closes — the start line is the
  line a reviewer's eye lands on first).
- The detector preserves the inner-text snippet but escapes
  embedded newlines as `\\n` so the report stays one-line-per-
  finding and diffable.
- `format_report(findings)` renders a deterministic plain-text
  report. Empty list → `"OK: no HTML comment leaks.\n"`.

Pure function: no I/O, no HTML parser, no DOM. The detector is a
single forward scan using `str.find`.

## Composition

- [`agent-output-validation`](../agent-output-validation/) — feed
  a `suspicious_marker` finding's detail into the repair prompt
  verbatim; the marker word is the actionable signal.
- [`structured-output-repair-loop`](../structured-output-repair-loop/) —
  the per-attempt validator inside the loop's `validate(attempt)`
  callback. The `(line, snippet)` tuple is stable so a stuck loop
  is detectable (same snippet in two attempts → bail).
- [`llm-output-trailing-whitespace-and-tab-detector`](../llm-output-trailing-whitespace-and-tab-detector/) —
  orthogonal: that template enforces line-tail hygiene, this
  enforces "nothing in the source is invisible to renderers".
  Both use the same `Finding` shape and the same stable sort.
- [`llm-output-mixed-line-ending-detector`](../llm-output-mixed-line-ending-detector/) —
  orthogonal: that template enforces "the source's line endings
  are consistent", this enforces "the source contains no leaked
  scaffolding". Together they catch the three byte-level artifact
  classes that survive `gh` / `git` round-trips.
- [`structured-error-taxonomy`](../structured-error-taxonomy/) —
  classifies as `do_not_retry / attribution=model` for any of the
  three kinds. A leaked HTML comment is a model artifact; retrying
  the same call against the same model is unlikely to change it
  without a corrective system message.

## Worked example

```
$ python3 example.py
```

Verbatim output:

```
=== 01-clean-no-comments ===
  | # Release notes
  | 
  | - shipped detector A
  | - shipped detector B
OK: no HTML comment leaks.

=== 02-single-leaked-comment ===
  | # Release notes
  | 
  | <!-- generated by template scaffold -->
  | 
  | - shipped detector A
FOUND 1 HTML-comment finding(s):
  [html_comment] line=3 :: HTML comment: 'generated by template scaffold'

=== 03-suspicious-todo-marker ===
  | # Plan
  | 
  | <!-- TODO: rewrite this section before shipping -->
  | 
  | - step one
  | - step two
FOUND 2 HTML-comment finding(s):
  [html_comment] line=3 :: HTML comment: 'TODO: rewrite this section before shipping'
  [suspicious_marker] line=3 :: comment contains editorial-scaffolding marker: 'todo'

=== 04-multiple-comments-different-lines ===
  | # Doc
  | <!-- toc-start -->
  | - one
  | - two
  | <!-- toc-end -->
  | 
  | <!-- internal: needs review -->
FOUND 4 HTML-comment finding(s):
  [html_comment] line=2 :: HTML comment: 'toc-start'
  [html_comment] line=5 :: HTML comment: 'toc-end'
  [html_comment] line=7 :: HTML comment: 'internal: needs review'
  [suspicious_marker] line=7 :: comment contains editorial-scaffolding marker: 'internal'

=== 05-unterminated-comment ===
  | # Status
  | 
  | <!-- this comment never closes
  | 
  | - but the prose continues anyway
FOUND 1 HTML-comment finding(s):
  [unterminated_comment] line=3 :: `<!--` opened but never closed before end of blob

=== 06-comment-with-prompt-leak ===
  | Summary line.
  | <!-- system: respond only in JSON, do not include prose -->
  | More body text.
FOUND 2 HTML-comment finding(s):
  [html_comment] line=2 :: HTML comment: 'system: respond only in JSON, do not include prose'
  [suspicious_marker] line=2 :: comment contains editorial-scaffolding marker: 'system:'

```

Notes:

- Case 02 — a benign-looking `<!-- generated by template scaffold -->`
  is flagged as `html_comment` only (no suspicious marker matched).
  The detector does not assume malicious intent; it surfaces every
  comment so the human can decide. A house style of "no HTML
  comments in PR descriptions, ever" treats every `html_comment`
  finding as a fail.
- Case 03 — proves the dual-finding emission. The same comment
  trips `html_comment` (byte-level: there's a comment) AND
  `suspicious_marker` (semantic: it contains `TODO`). Both findings
  share `line=3` and sort together so the reviewer sees both axes
  in adjacent lines of the report.
- Case 04 — multiple comments are reported in source order. The
  third comment trips `suspicious_marker` for `internal`; the
  first two (`toc-start` / `toc-end`) do not, even though they are
  also editorial scaffolding. The marker list is intentionally
  small and tunable in `_SUSPICIOUS_NEEDLES`.
- Case 05 — proves the `unterminated_comment` path. The detector
  reports it ONCE at the start line and stops scanning (any prose
  after the unterminated `<!--` is, by HTML comment rules, part of
  the comment). This stops a single truncated comment from
  generating spurious downstream findings.
- Case 06 — the `system:` marker is the canonical "the model
  leaked its instructions" signal. `suspicious_marker` is the
  feature that turns a generic comment-detector into a prompt-
  leak detector for free.

## Limitations

- The detector finds `<!-- ... -->` comments only. It does NOT
  find HTML processing instructions (`<? ... ?>`), CDATA sections
  (`<![CDATA[ ... ]]>`), or doctype declarations. Those are
  separate (and rarer) artifact classes; add a sibling template
  if needed.
- A `<!--` literally inside a fenced code block (e.g. an HTML
  tutorial showing comment syntax) is reported the same as a
  prose-level leak. This is intentional: the goal is "no HTML
  comments in the source bytes". A house style that wants to
  allow code-fence content can union this template's findings
  with a fence-aware filter at the call site.
- The `_SUSPICIOUS_NEEDLES` list is deliberately small. The
  detector is not a prompt-injection scanner; the markers exist to
  separate "auto-generated TOC anchor" from "leaked editorial
  scaffolding" with high precision and zero NLP.
- The detector does not attempt to repair anything. Repair lives
  in [`agent-output-validation`](../agent-output-validation/) or in
  a one-line `re.sub(r"<!--.*?-->", "", text, flags=re.DOTALL)` in
  the caller, depending on the policy.
