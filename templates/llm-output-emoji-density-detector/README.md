# `llm-output-emoji-density-detector`

Pure stdlib detector for emoji density and emoji clustering in an
LLM Markdown / prose output blob — the most legible "AI tell" in
any document that lands in a permanent record (`git log`,
`gh pr view`, a wiki, a status channel). Editors do not strip
emoji on save; a draft generated by an assistant carries every
sparkle through to commit.

Three finding kinds:

- `per_line_dense` — a single line carries more emoji than the
  configured per-line cap (default `3`). Reported once per line at
  the column of the first offending emoji.
- `cluster` — two emoji appear back-to-back, separated by zero
  characters or a single ASCII space. Back-to-back emoji are the
  strongest visual tell of a "cheerful assistant" register and
  almost never appear in human-written prose outside of social-
  media bios. Reported per pair, so a run of 5 emoji yields 4
  cluster findings.
- `density_over_cap` — document-level: total emoji per 100
  whitespace-split words exceeds the configured cap (default
  `5.0`). Reported once at line=0, col=0.

A multi-codepoint emoji (family glyph, profession with skin tone,
etc.) joined by ZWJ (U+200D) and U+FE0F counts as **one** emoji,
matching how the user sees it. See case 05 in the worked example.

## When to use

- Pre-publish gate on any LLM-generated **commit message body**,
  **PR description**, or **issue body** before `gh` / `git`
  writes it to a permanent record. A `cluster` finding is the
  single most legible signal that the body was not lightly
  edited; flagging it forces a human pass.
- Pre-flight on **status reports** and **runbooks** drafted by an
  assistant before they are pasted into a team channel or wiki.
  The `per_line_dense` cap of 3 is conservative for technical
  audiences; bump it for marketing-flavored channels.
- Audit step in a **review-loop** that promotes
  [`agent-output-validation`](../agent-output-validation/)'s
  `repair_once` policy: a `cluster` finding feeds the line back
  into the repair prompt with a fixed instruction ("delete the
  emoji at column N").
- Cron-friendly: findings are sorted by `(line_number, kind,
  column)` and the report is rendered in a deterministic format,
  so byte-identical output across runs makes diff-on-the-output a
  valid CI signal.

## Inputs / outputs

```
detect_emoji_issues(
    text: str,
    *,
    per_line_cap: int = 3,
    per_100_words_cap: float = 5.0,
) -> list[Finding]

Finding(kind: str, line_number: int, column: int, raw: str, detail: str)
```

- `text` — the LLM output to scan. Must be `str` (raises
  `ValidationError` otherwise).
- `per_line_cap` — max emoji per line before `per_line_dense`
  fires. Must be >= 1.
- `per_100_words_cap` — max emoji per 100 whitespace-split words
  before `density_over_cap` fires. Must be > 0.
- `Finding.line_number` is 1-based; `Finding.column` is 1-based.
  Document-level findings use `(0, 0)`.
- `format_report(findings)` renders a deterministic plain-text
  report. Empty list → `"OK: emoji density within caps.\n"`.

Pure function: no I/O, no Markdown parser, no Unicode database
lookup. The detector uses Unicode block ranges that have been
stable since Unicode 6.0; new 2024+ emoji that land outside these
blocks are not flagged, but density is the issue, not coverage.

## Composition

- [`agent-output-validation`](../agent-output-validation/) — feed
  a `cluster` finding's `(line_number, column, kind)` into the
  repair prompt for a one-turn fix.
- [`structured-output-repair-loop`](../structured-output-repair-loop/) —
  the per-attempt validator inside the loop's `validate(attempt)`
  callback; the `(line_number, column, kind)` tuple makes a stuck
  loop detectable (same tuple twice in a row → bail).
- [`llm-output-trailing-whitespace-and-tab-detector`](../llm-output-trailing-whitespace-and-tab-detector/) —
  orthogonal: that template enforces line-ending hygiene, this
  enforces emoji discipline. Both use the same `Finding` shape
  and a stable sort, so unioned reports stay diffable.
- [`structured-error-taxonomy`](../structured-error-taxonomy/) —
  classifies as `do_not_retry / attribution=model` for any of the
  three kinds. Emoji density is a register choice the model made;
  retrying the same prompt against the same model is unlikely to
  change it without a corrective system message.

## Worked Example Output

```
$ python3 example.py
=== 01-clean ===
input:
  | Daily summary:
  | - shipped two templates.
  | - ran the linter and fixed two findings.
  | - queued the next mission.
OK: emoji density within caps.

=== 02-cluster-pair ===
input:
  | Release notes:
  | - the migration finished 🎉🚀 ahead of schedule.
  | - one minor regression filed against the parser module.
FOUND 2 emoji finding(s):
  [density_over_cap] doc-level :: document density 11.11 emoji per 100 words (cap=5.0, total_emoji=2, words=18)
  [cluster] line=2 col=26 :: adjacent emoji at columns 26 and 27 (gap='')
    line='- the migration finished 🎉🚀 ahead of schedule.'

=== 03-per-line-over-cap ===
input:
  | Weekly highlights:
  | Wins: 💪 🎉 🚀 ✨ 🔥 huge week!
  | Plans: ship the dashboard.
FOUND 6 emoji finding(s):
  [density_over_cap] doc-level :: document density 55.56 emoji per 100 words (cap=5.0, total_emoji=5, words=9)
  [cluster] line=2 col=7 :: adjacent emoji at columns 7 and 9 (gap=' ')
    line='Wins: 💪 🎉 🚀 ✨ 🔥 huge week!'
  [cluster] line=2 col=9 :: adjacent emoji at columns 9 and 11 (gap=' ')
    line='Wins: 💪 🎉 🚀 ✨ 🔥 huge week!'
  [cluster] line=2 col=11 :: adjacent emoji at columns 11 and 13 (gap=' ')
    line='Wins: 💪 🎉 🚀 ✨ 🔥 huge week!'
  [cluster] line=2 col=13 :: adjacent emoji at columns 13 and 15 (gap=' ')
    line='Wins: 💪 🎉 🚀 ✨ 🔥 huge week!'
  [per_line_dense] line=2 col=7 :: 5 emoji on this line (cap=3)
    line='Wins: 💪 🎉 🚀 ✨ 🔥 huge week!'

=== 04-doc-density-over-cap ===
input:
  | ok 🎉 done 🚀 next ✨ cool 🔥
FOUND 2 emoji finding(s):
  [density_over_cap] doc-level :: document density 100.00 emoji per 100 words (cap=5.0, total_emoji=4, words=4)
  [per_line_dense] line=1 col=4 :: 4 emoji on this line (cap=3)
    line='ok 🎉 done 🚀 next ✨ cool 🔥'

=== 05-zwj-joined-counts-as-one ===
input:
  | Family photo: 👨‍👩‍👧 attached.
  | No other emoji on this line so no cluster finding.
FOUND 1 emoji finding(s):
  [density_over_cap] doc-level :: document density 7.14 emoji per 100 words (cap=5.0, total_emoji=1, words=14)

=== 06-emoji-then-comma-then-emoji ===
input:
  | Recap: 🎉, and later 🚀 launched cleanly.
FOUND 1 emoji finding(s):
  [density_over_cap] doc-level :: document density 33.33 emoji per 100 words (cap=5.0, total_emoji=2, words=6)
```

Notes:

- Case 02 — two adjacent emoji separated by zero characters trip
  one `cluster` finding. The density is over cap because the
  blob is small (only 18 words); the per-line cap of 3 is not
  exceeded. The cluster finding is the actionable one — the fix
  is "delete one of the two emoji".
- Case 03 — five space-separated emoji on the same line trip
  four `cluster` findings (one per adjacent pair) plus one
  `per_line_dense` finding at the line level. Sorted output puts
  the document-level finding first, then per-line findings in
  column order, so a reviewer reads top-down without re-sorting.
- Case 04 — single line, single paragraph, density at 100 emoji
  per 100 words. The per-line cap fires; cluster does not (the
  emoji are separated by full words, not just a space).
- Case 05 — proves the ZWJ-joined cluster invariant. The family
  glyph `👨‍👩‍👧` is one emoji, not three, so neither `per_line_dense`
  nor `cluster` fires. Only the document-level density tips
  because the blob is small. Bump `per_100_words_cap` to `10.0`
  to suppress that finding while still catching real abuse.
- Case 06 — proves the cluster gap rule. Two emoji separated by
  `", and later "` are not a cluster; only the doc-level density
  finding fires (and it can be tuned away).

## Limitations

- Emoji recognition uses Unicode block ranges that have been
  stable since Unicode 6.0. Post-2024 emoji that land outside
  these blocks are not flagged. Density is the issue this
  template addresses, not coverage; a single missed glyph does
  not change the verdict.
- Word count is whitespace-split. CJK text without spaces will
  read as a single "word" and inflate the density ratio. For
  CJK-heavy output, set `per_100_words_cap` to a much higher
  value (e.g. `50.0`) and rely on `cluster` and `per_line_dense`.
- Variation selector U+FE0F and ZWJ U+200D are handled, but
  skin-tone modifiers (U+1F3FB..U+1F3FF) are NOT explicitly
  joined. They render correctly because the modifier follows the
  base emoji and is not itself in an emoji range, so it does not
  get re-counted; effectively a skin-toned emoji counts as one
  emoji even without explicit handling.
- The detector does not distinguish between Markdown contexts
  (heading vs. body vs. code fence). Emoji in fenced code blocks
  count toward density. If you intentionally include an emoji in
  a code sample, raise `per_line_cap` for that document or
  preprocess the blob to strip fences before scanning.
